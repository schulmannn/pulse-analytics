# Память проекта Atlavue

Обновлено: 2026-07-16.

Это единственный живой документ о текущем состоянии продукта, принятых решениях и ближайшей
очереди. История изменений остаётся в git и PR: сюда не переносятся завершённые аудиты, спринты и
подробные бэклоги. После каждого релиза файл сокращается и обновляется, а не превращается в changelog.

## Продукт

Atlavue — мульти-источниковая аналитика Telegram и Instagram для авторов и команд. Продукт
объединяет метрики аккаунтов и публикаций, кампании из постов разных источников, отчёты и
настраиваемую Главную с виджетами.

Telegram подключается двумя способами:

- управляемое подключение по QR: `StringSession` хранится в Postgres только в зашифрованном виде
  (AES-256-GCM, ключ `TG_SESSION_KEY`) и используется приватным MTProto-сервисом;
- локальный collector: сессия остаётся на устройстве пользователя, Atlavue получает только
  производные метрики через идемпотентный ingest.

Legacy `TG_SESSION` внутри MTProto-сервиса остаётся fallback для центрального канала. Основной
credential его владельца хранится как зашифрованная managed QR-сессия; дневной `velocity` уже
считается через неё и сохраняется DB-first, а `post_stats` и public media пока остаются отдельными
последующими security-срезами. Живой поиск
упоминаний уже работает через managed-сессию вызывающего пользователя.

## Текущая архитектура

- `frontend/` — React 18, Vite, TypeScript strict, TanStack Query, Zod и Tailwind. Сборка
  `frontend/dist` отдаётся Express на `/`; прежний интерфейс доступен только как временный
  fallback на `/legacy`.
- `server/` — Node.js/Express. Путь запуска: `config.js` -> `app.js` -> `composition.js` ->
  `main.js`, тонкий entrypoint — `index.js`. Доменные зависимости идут через
  `routes/` -> `services/` -> `repos/` -> `db/`; фоновые задачи находятся в `jobs/`.
- `server/migrations/` — forward-only SQL-миграции, которые применяются перед запуском web.
- `mtproto/service.py` — приватный FastAPI/Telethon-сервис; web обращается к нему по
  `MTPROTO_URL` с общим `MTPROTO_TOKEN`.
- `collector/pulse_collector.py` — альтернативный локальный агент с SQLite-очередью и retry.
- Postgres — постоянное хранилище пользователей, workspaces, источников, метрик, кампаний,
  отчётов, интеграций и служебных job-state.

## Неподвижные инварианты

- Workspace — граница доступа. Любой tenant-read/write проходит ownership/role-check и несёт
  `channel_id` либо другой явный workspace-scoped идентификатор.
- Внешний источник имеет каноническую identity в `external_sources`; повторное подключение не
  должно создавать независимые наборы данных одного аккаунта или канала.
- QR-сессии Telegram и Instagram-токены шифруются до записи в БД и никогда не попадают в логи,
  API-ответы, экспорт GDPR или git.
- Managed Telegram-сессии всегда записываются активным `TG_SESSION_KEY`; во время ротации разрешено до трёх
  read-only ключей в `TG_SESSION_KEY_PREVIOUS`. Успешное fallback-чтение лениво перешифровывает только ciphertext
  под тем же `session_version`, поэтому конкурентное переподключение не может быть затёрто. Instagram-ключи
  остаются отдельным активным контуром без этого fallback.
- Правила Telegram-упоминаний принадлежат выбранному `channel_id`: канал без `include_terms` не
  запускает поиск, а отсутствие настроек никогда не подменяется клиентским или серверным seed.
  Читать правила может участник с доступом к каналу; изменять их и расходовать поисковую квоту —
  только owner/admin. Live-поиск использует исключительно managed QR-сессию вызывающего пользователя.
- Здоровье управляемой Telegram QR-сессии хранится отдельно от свежести аналитики. Только точная
  auth-ошибка переводит подключение в `reauth_required`; временные upstream-сбои получают
  `degraded` и не требуют повторного входа. Переподключение заменяет только зашифрованную сессию,
  сохраняет каналы и историю и сразу запускает обновление собственных central/QR-каналов. Каналы,
  доступные через workspace, но принадлежащие другому пользователю, этим credential не обновляются.
- Приватный managed Telegram-канал после первого успешного resolve хранит account-scoped `access_hash`
  как BIGINT и передаёт его в MTProto только decimal string. Warm-сбор не сканирует диалоги; legacy/stale
  identity делает не более одного cold resolve. Читать/обновлять hash может только внутренний сбор владельца
  на текущей `session_version`; reconnect, чужой workspace credential и entity mismatch fail closed. Поле не
  входит в channel API, snapshots, audit или логи, а публичные username-каналы его вообще не используют.
- Legacy `source='central'` не конвертируется в QR-канал и не получает новый `channel_id` при
  переподключении. Daily ingest предпочитает managed-сессию владельца, пишет в прежний архив и
  использует global MTProto только как fallback. Velocity считается общей реализацией на уже
  загруженных постах, включается только для central collect, укладывается в общий 110-секундный
  бюджет и сохраняется атомарно со snapshot/daily/posts; обычные QR-каналы этот fan-out не запускают.
- TG views и IG reach — разные величины. Их нельзя незаметно суммировать или показывать под одним
  общим названием.
- В Telegram «Просмотры канала» — сумма дневного потока `channel_daily.views`, включая просмотры
  ранее опубликованного контента. «Просмотры публикаций» — сумма просмотров только тех постов,
  которые опубликованы в выбранном окне. Эти показатели подписываются явно и не обязаны совпадать.
- Выбранный источник стабилен при навигации. Маршрут конкретной сети (`/instagram`, `/posts` и
  другие) становится последним явным выбором; общие маршруты (`/home`, `/reports`, кампании) его
  сохраняют даже после полной перезагрузки. Загрузка данных и порядок источников в API не меняют
  этот выбор.
- `main` автоматически разворачивается на Railway. Изменения выпускаются отдельным PR после
  зелёного CI; worker-агенты не коммитят и не выпускают, интеграцию выполняет Codex или человек.
- CI запускает backend unit/integration tests с Postgres, Python tests/compile, frontend tests,
  typecheck/build, bundle gate и Playwright smoke на детерминированных данных.
- Web-процесс не ждёт Postgres бесконечно: базовый пул ограничен 10 соединениями, connection/statement/query
  deadlines заданы явно, а `query_timeout` обязан быть больше `statement_timeout`. Парольный `scrypt` выполняется
  асинхронно и не блокирует event loop.
- Авторизация не кэширует user/status/token_version: отзыв сессии остаётся немедленным. При выборе канала по
  умолчанию tenant middleware получает только первый доступный `channel_id`, не вычисляя продуктовые
  `memberCount` и `ig_connected` для всего списка; итоговый доступ и effective role по-прежнему проверяет
  actor-scoped `getChannel`.
- Долгий browser-cache разрешён только для Vite assets с content hash. HTML, SPA fallback и unhashed assets всегда
  ревалидируются; HTTP-ответы сжимаются на уровне Express.
- Тяжёлые collection/report/maintenance jobs используют отдельный Postgres pool (дефолт 2) и не занимают live
  HTTP/auth/tenant pool. Instagram возобновляется по account identity/day, Telegram QR — по channel/day;
  single-flight recovery runner добирает незавершённые bounded-проходы после рестарта и останавливается до drain.
- In-memory response cache использует bounded LRU с абсолютным TTL: чтение повышает recency, но не продлевает
  срок жизни. Cap и TTL задаются валидированными `CACHE_MAX_ENTRIES` / `CACHE_TTL_MS`.
- MTProto live-запросы и background collection имеют независимые circuit states и half-open recovery, поэтому
  сбои фонового канала не открывают circuit живого дашборда. Один общий in-flight bulkhead по-прежнему защищает
  единый upstream-процесс от суммарной перегрузки.
- Instagram Graph повторяет только transient versioned data GET (не более двух bounded backoff) внутри общего
  singleflight. Rate limits сохраняют `Retry-After` и безопасные Graph-коды; OAuth, token exchange и refresh
  остаются single-shot и никогда не попадают в этот retry-loop.
- Instagram OAuth callback допускает не более восьми одновременных последовательных обменов и восьми ожидающих
  запросов на единственной web-реплике; ожидание ограничено двумя секундами, после чего UI получает честный `busy`.
  Подписанный state, active-user и актуальная admin/owner-роль проверяются до admission и обращения к провайдеру.
  Telegram QR-cap также считается защитным backpressure: он не открывает MTProto circuit, не означает reauth и
  возвращает `Retry-After`; frontend не усиливает такой пик автоматическим повтором.
- Scheduled report email имеет durable at-most-once reservation на период и стабильный Resend
  `Idempotency-Key`. 429 освобождает точную reservation для следующего daily retry; timeout/5xx и crash после
  reservation считаются неоднозначными и fail closed: возможен один пропуск, но не автоматический дубль.
- Ежедневная maintenance на отдельном background pool ограниченно удаляет только старые terminal
  `jobs` и уже недействующие `email_tokens`: дефолт 30 дней, батчи до 500 × 40, детерминированный порядок и
  `FOR UPDATE SKIP LOCKED`. `queued`/`running`, валидные неиспользованные токены, аналитика, ingest receipts,
  audit trail, bugs и attachments этим контуром не удаляются.
- `uncaughtException` и `unhandledRejection` являются фатальными: web перестаёт принимать трафик, использует
  единый bounded drain и завершается с кодом 1; принудительный таймер не позволяет зависшему shutdown оставить
  процесс в неизвестном состоянии. Клиентские crash-события атомарно обновляют ledger и один видимый admin-тикет
  на сигнатуру; повторы сохраняют ручной статус тикета, обновляют последний trace и показывают честный счётчик.

## Продуктовые и дизайн-решения

- Сайдбар остаётся плоским и не размножает пункты по каждому источнику. Основной контекст задаёт
  один глобальный переключатель источника; на отдельных рабочих поверхностях допустимы
  контекстные фильтры, например точный источник внутри кампании.
- Текущий приоритет — desktop и тёмная тема. Мобильный редизайн выполняется только после
  завершения desktop-программы; существующее мобильное поведение нельзя ломать по пути.
- Steep — ориентир по плотности, иерархии и взаимодействиям, а не шаблон для пиксельного копирования.
- Цвет графика обозначает серию/идентичность; зелёный и красный зарезервированы для оценённого
  положительного или отрицательного изменения. Сравнение с предыдущим периодом обязательно там,
  где оно методологически корректно.
- Пользовательские размеры виджетов называются `S`, `M`, `L` и соответствуют 33%, 50% и 100% шестиколоночной
  сетки. Внутренние значения `third`, `half`, `full` остаются стабильными для совместимости с сохранёнными раскладками.
  Временной line chart не уменьшается до `S`, если при этом перестают читаться ось и динамика.
- На «Обзоре» Telegram и Instagram агрегат «Показатели» разделён на пять самостоятельных KPI-карточек: две `M`
  для главных сигналов и три `S` для компактных сравнений. Все карточки читают один page-period. Старые составные ключи `kpi` и
  `ig-kpi` сохранены только для совместимости с уже настроенными пользовательскими «Главными» и не мигрируются автоматически.
- На source/work-вкладках Telegram и Instagram существует один авторитетный период в top bar:
  `7д` / `30д` / `90д` / `Всё` / `Свой период`. Датируемые карточки, таблицы и сравнения читают это
  окно; локальные period controls внутри карточек не показываются. Bare archive keys `YYYY-MM-DD`
  трактуются как локальные календарные дни без UTC-сдвига. Единственное исключение — Главная.
- Desktop-экспорт Аналитики Telegram/Instagram использует общую long-form CSV-схему и подписывает
  каждую строку точными границами `current` или равного `previous`-окна; aggregate-only метрики не
  превращаются в дневные точки. Экспорт Контента выгружает ровно текущие загруженные строки таблицы
  после периода, кампании, поиска, формата и сортировки и не обещает полный исторический архив.
  Отсутствующая метрика остаётся пустой, реальный ноль сохраняется; имена файлов содержат источник и
  окно, а пользовательские строки нейтрализуются от spreadsheet-formula injection.
- Главная — персональная доска, а не ещё один доменный дашборд. Каждый виджет сохраняет собственные
  платформу, источник, период и настройки; глобальный переключатель не переписывает его identity.
  Автосборка применяется только к пустой desktop-доске и учитывает доступность Telegram/Instagram;
  существующие доски не мигрируются и не переупорядочиваются.
- Упоминания читаются из сохранённого архива. Живой Telegram-поиск только пополняет архив и не
  подменяет его в UI; «потенциальные просмотры» означают сумму просмотров упомянувших постов без
  дедупликации аудитории и не называются охватом. Для каждого Telegram-канала отдельно хранятся
  включающие термины, исключения по тексту/источнику и режим `contains`/`word`; собственный канал
  исключается автоматически. Диакритические варианты остаются отдельными запросами Telegram,
  потому что API не гарантирует accent-folding глобального поиска.
- Отчёт — сохранённый read-first документ, а не постоянно открытый конструктор. Изменения desktop-
  версии накапливаются в локальном draft и отправляются одним PUT только по явному «Сохранить»;
  доставка письмом содержит ссылку на пересчитываемый документ, а PDF создаётся через печать.
- Telegram-отчёт закрепляет собственный источник и явно подписывает его независимо от глобального
  переключателя. Drill-down сначала активирует этот Telegram-источник, чтобы следующая страница не
  показывала данные другого канала.

## Состояние production

Последний production baseline: PR #217 — managed-central Telegram velocity считается через
owner-scoped сессию и сохраняется тем же DB-first путём без зависимости от legacy credential.

Недавно выпущено:

- #217 — общая реализация velocity используется legacy и managed путями; central collect явно
  включает fan-out, повторно использует уже загруженные посты и атомарно сохраняет доступный payload,
  а обычные QR-каналы не платят за дополнительные RPC;
- #216 — Analytics CSV использует общую long-form схему current/previous, Content CSV повторяет
  текущий post-filter/sort row set; filenames window/source-scoped, missing metrics и CSV formulas
  обрабатываются безопасно, неизвестные TG graph series не переименовываются по позиции;
- #215 — Instagram OAuth callback имеет конечные in-flight/queue/deadline, повторно проверяет write-role до
  провайдера; Telegram QR-cap не открывает circuit, возвращает `Retry-After`, UI не усиливает его автоповтором;
- #214 — private Telegram entity identity сохраняется без потери int64, warm path не сканирует диалоги,
  stale hash self-heal bounded; reconnect/tenant/entity guards и отсутствие утечек покрыты тестами;
- #213 — default-channel resolution больше не материализует `listChannels` с `memberCount`/`ig_connected`;
  видимость, disabled-фильтр, порядок, effective role и немедленный session revoke сохранены;
- #212 — bounded daily cleanup для terminal `jobs` и мёртвых `email_tokens`, partial jobs index,
  `SKIP LOCKED`, конфигурируемый горизонт и реальные PG-тесты границ/cap/concurrency;

Декомпозиция `server/index.js` и `server/db.js` достаточна для текущего продуктового этапа.
Оставшиеся архитектурные улучшения отложены и не блокируют редизайн.

## Активная очередь

1. При наличии production DB evidence проверить реальные query plans и исправить только доказанные
   SQL/index bottlenecks.
2. Проверить production sizing и закрыть только подтверждённые BIGINT-границы; отдельно принять продуктовые
   сроки хранения для canonical analytics, ingest receipts и audit trail — operational retention уже ограничен.

## Известные ограничения и риски

- Разрешена одна web-реплика: `WEB_REPLICAS > 1` отклоняется до появления shared runtime state.
- Recovery collection пока исполняется внутри web-процесса, хотя использует отдельный DB pool и durable item-level
  leases. Отдельный Railway worker потребуется как следующий инфраструктурный рубеж при росте CPU/network нагрузки.
- `/qr/collect` пока использует short-lived Telethon client на один сбор. Сохранённый `access_hash` убирает
  полный dialog scan; bounded reuse клиентов требует отдельного lifecycle/eviction дизайна, чтобы не держать
  сотни пользовательских сессий и соединений бесконтрольно.
- Telegram `searchPosts` для упоминаний имеет жёсткую квоту и фактически требует Premium; поиск
  запускается только явным действием пользователя.
- Управляемая Telegram `StringSession` даёт широкий доступ к аккаунту и остаётся наиболее
  чувствительным секретом системы. После ротации прежний ключ нельзя удалять, пока нужные сессии не были
  успешно использованы и лениво перешифрованы; отдельного eager rotation job и key-id в ciphertext пока нет.
- `post_stats` и public media центрального Telegram-канала пока могут оставаться недоступными при
  протухшей global `TG_SESSION`; velocity уже использует owner-scoped managed путь.
- Миграции forward-only. Перед изменением схемы нужен snapshot; восстановление описано в
  `ops/BACKUP_RESTORE.md`.
- Frontend bundle ограничен CI-гейтом: новые зависимости добавляются только при доказанной пользе.
- В отложенной светлой теме active-period chip пока даёт контраст 4.22:1 вместо 4.5:1; в мобильной
  ветке period controls имеют высоту 22px вместо целевых 32px. Оба долга зафиксированы тестами и не
  исправляются внутри завершённой desktop dark-программы.

## Как обновлять память

После каждого production-релиза:

1. обновить продуктовый baseline и оставить не более пяти последних крупных релизов;
2. удалить завершённый пункт из активной очереди и уточнить следующий;
3. зафиксировать только новые долгоживущие решения, инварианты или риски;
4. удалить устаревший текст вместо добавления исторической сноски.
